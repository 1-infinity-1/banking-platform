# C4 Диаграммы

C4 Model — иерархия из четырёх уровней абстракции: Context → Container → Component → Code. Диаграммы ниже охватывают первые три уровня.

Все диаграммы выполнены на `mermaid flowchart` с единым визуальным языком. Канонический синтаксис `C4Context / C4Container / C4Component` сознательно не используется: его рендерер в GitHub Markdown даёт скученный layout и пересечения связей. Семантика C4 сохраняется через `subgraph` и согласованные `classDef`.

## Легенда нотации

Цветовая палитра и формы узлов одинаковы во всех диаграммах ниже — это позволяет читать диаграммы по визуальному коду, не вдаваясь в подписи.

```mermaid

    sys["🏦 System (root)"]:::system

    comp["Component (модуль)"]:::component

    extSkel[/"External [скелет]"/]:::skeletonExt
    db[("Database (cylinder)")]:::database

    sync1["A"]:::component
    sync2["B"]:::component
    async1["C"]:::component
    async2["D"]:::component

    sync1 -- "REST / gRPC (sync)" --> sync2
    async1 -. "async / event (Kafka)" .-> async2

    classDef person      fill:#E3F2FD,stroke:#1565C0,stroke-width:2px,color:#0D47A1
    classDef system      fill:#FFF3E0,stroke:#E65100,stroke-width:2px,color:#BF360C
    classDef container   fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeleton    fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
    classDef component   fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B

    classDef skeletonExt fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
    classDef database    fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
    classDef infra       fill:#F3E5F5,stroke:#6A1B9A,stroke-width:2px,color:#4A148C
```

| Класс | Цвет | Когда применяется |
|---|---|---|
| `person` | синий (round shape) | Конечные пользователи и операторы (внешние акторы). |
| `system` | оранжевый | Корневая система (Banking Platform) — только на Level 1. |
| `container` | зелёный | Реализованный исполняемый юнит (сервис, БД, broker). |
| `skeleton` | серый, пунктирная рамка | Сервис со структурой кода, но без бизнес-логики (`[скелет]`). |
| `component` | голубой | Модуль внутри сервиса на Level 3. |
| `external` | зелёный (parallelogram) | Внешний сервис, на который ссылается текущая диаграмма Level 3. |
| `skeletonExt` | серый пунктир (parallelogram) | То же, но для скелетных сервисов. |
| `database` | жёлтый цилиндр | PostgreSQL-инстанс. |
| `infra` | фиолетовый hex | Инфраструктура (Kafka). |

**Стрелки:**

- `-->` сплошная — синхронный вызов (REST, gRPC, SQL).
- `-.->`  пунктирная — асинхронный путь (Kafka produce/consume), а также путь «on error» внутри компонента.

## Уровень 2 — Containers

Какие исполняемые единицы входят в систему, как они общаются и где хранят данные.

```mermaid
flowchart TB
    customer(["👤 Клиент"]):::person
    admin(["👤 Администратор"]):::person

    subgraph banking["🏦 Banking Platform"]
        direction TB

        subgraph edge["Edge"]
            gateway["<b>Gateway Service</b><br/><span style='font-size:11px'>Go · ogen/OpenAPI<br/>REST API · :8081</span>"]:::container
        end

        subgraph auth_sub["Auth"]
            auth["<b>Auth Service</b><br/><span style='font-size:11px'>Go · gRPC · :8082<br/>JWT · сессии · роли</span>"]:::container
            authDb[("Auth DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>users · sessions · devices<br/>refresh_tokens · roles · permissions</span>")]:::database

            ledgerConsumer["<b>Ledger Consumer</b><br/><span style='font-size:11px'>Go · Kafka consumer<br/>Создаёт проводки</span>"]:::container
            ledgerDb[("Ledger DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>ledger_entries</span>")]:::database

    %% External REST
    customer -- "REST :8081" --> gateway
    admin    -- "REST :8081" --> gateway

    %% Internal gRPC (sync)
    gateway -- "gRPC :8082" --> auth
    gateway -- "gRPC :8083" --> account

    classDef infra     fill:#F3E5F5,stroke:#6A1B9A,stroke-width:2px,color:#4A148C
```

**Чтение:**

- **Edge** — единственная точка входа (gateway). Все REST-запросы идут только сюда.
- **Auth / Account / Transaction / Ledger** — четыре доменных сервиса, каждый со своей БД (database-per-service).
- **`[скелет]`** на блоках означает, что код есть как структура (`internal/<service>/`), но бизнес-логика ещё не реализована.
- **Сплошные стрелки** — синхронный вызов (REST/gRPC). **Пунктирные** — асинхронные сообщения через Kafka.
- **gateway → transaction → account** — единственный кросс-доменный gRPC-вызов на уровне платформы (нужен для атомарного списания/зачисления в переводах).

## Уровень 3 — Components: Gateway Service

Из каких компонентов состоит `gateway-service` — самый сложный сервис с fan-out к остальным.

```mermaid
flowchart TB
    client(["👤 Клиент"]):::person

    subgraph gw["Gateway Service · :8081"]
        direction TB

        subgraph mw["HTTP Middleware (ingress chain)"]
            direction LR
            traceMW["Trace MW<br/><span style='font-size:11px'>x-trace-id · x-request-id<br/>→ context.Context</span>"]:::component
            logMW["Logging MW<br/><span style='font-size:11px'>method · path · status · latency</span>"]:::component
        end

        handlers["HTTP Handlers (ogen)<br/><span style='font-size:11px'>Сгенерированы из openapi.yaml<br/>валидация → domain types</span>"]:::component
        services["Services<br/><span style='font-size:11px'>Бизнес-оркестрация:<br/>выбор gRPC-клиента · маппинг ответов</span>"]:::component
        errHandler["Error Handler<br/><span style='font-size:11px'>GatewayHandler.NewError<br/>domain error → HTTP status</span>"]:::component

        subgraph clients["gRPC Clients (с trace metadata)"]
            direction LR
            authClient["Auth Client"]:::component
            accountClient["Account Client"]:::component
            txClient["Transaction Client"]:::component
            ledgerClient["Ledger Client"]:::component
        end
    end

    auth[/"Auth Service<br/>gRPC :8082"/]:::external
    account[/"Account Service<br/>gRPC :8083 [скелет]"/]:::skeletonExt
    tx[/"Transaction Service<br/>gRPC :8084 [скелет]"/]:::skeletonExt
    ledger[/"Ledger Service<br/>gRPC :8085"/]:::external

    client     -- "HTTP/JSON" --> traceMW
    traceMW    --> logMW
    logMW      --> handlers
    handlers   -- "domain types" --> services
    handlers   -. "on error" .-> errHandler

    services --> authClient
    services --> accountClient
    services --> txClient
    services --> ledgerClient

    authClient    -- "gRPC + metadata" --> auth
    accountClient -- "gRPC + metadata" --> account
    txClient      -- "gRPC + metadata" --> tx
    ledgerClient  -- "gRPC + metadata" --> ledger

    classDef person      fill:#E3F2FD,stroke:#1565C0,stroke-width:2px,color:#0D47A1
    classDef component   fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external    fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeletonExt fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
```

**Чтение:**

- HTTP-запрос идёт через цепочку middleware `Trace → Logging → Handlers (ogen)`.
- Handlers вызывают внутренние Services, которые оркестрируют выбор нужного gRPC-клиента.
- Все 4 gRPC-клиента (`authClient`, `accountClient`, `txClient`, `ledgerClient`) инжектируют `x-trace-id` / `x-request-id` в metadata — это даёт сквозную трассировку.
- Ошибки уходят через `Error Handler` (пунктирная стрелка), который маппит domain-ошибки в HTTP статусы.

## Уровень 3 — Components: Auth Service

Внутренняя структура `auth-service` — наиболее полностью реализованного сервиса.

```mermaid
flowchart TB
    gateway[/"Gateway Service<br/><span style='font-size:11px'>gRPC client</span>"/]:::external

    subgraph authSvc["Auth Service · :8082"]
        direction TB

        interceptors["gRPC Interceptors<br/><span style='font-size:11px'>chain: trace → logging → error mapping → recovery</span>"]:::component

        subgraph transport_layer["Transport (gRPC)"]
            direction LR
            authTransport["AuthTransport<br/><span style='font-size:11px'>Login · Logout · RefreshToken</span>"]:::component
            mgmtTransport["ManagementTransport<br/><span style='font-size:11px'>CreateUser</span>"]:::component
        end

        subgraph service_layer["Services (business logic)"]
            direction LR
            authService["AuthService<br/><span style='font-size:11px'>bcrypt · session lifecycle<br/>token generation</span>"]:::component
            mgmtService["ManagementService<br/><span style='font-size:11px'>создание пользователей<br/>назначение ролей</span>"]:::component
        end

        subgraph crosscut["Cross-cutting"]
            direction LR
            tokenMgr["JWT Manager<br/><span style='font-size:11px'>golang-jwt<br/>access/refresh</span>"]:::component
            txManager["Tx Manager<br/><span style='font-size:11px'>pgx<br/>BeginFunc(ctx, fn)</span>"]:::component
        end

        subgraph repos["Repositories (pgx)"]
            direction LR
            userRepo["User Repo"]:::component
            sessionRepo["Session Repo"]:::component
            deviceRepo["Device Repo"]:::component
            tokenRepo["RefreshToken Repo"]:::component
        end
    end

    db[("Auth DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>users · sessions · devices<br/>refresh_tokens · roles · permissions</span>")]:::database

    gateway       -- "gRPC" --> interceptors
    interceptors  --> authTransport
    interceptors  --> mgmtTransport
    authTransport -- "domain types" --> authService
    mgmtTransport -- "domain types" --> mgmtService

    authService   -- "generate/validate" --> tokenMgr
    authService   -- "BeginFunc" --> txManager
    mgmtService   -- "BeginFunc" --> txManager

    txManager -- "pgx.Tx" --> userRepo
    txManager -- "pgx.Tx" --> sessionRepo
    txManager -- "pgx.Tx" --> deviceRepo
    txManager -- "pgx.Tx" --> tokenRepo

    userRepo    -- "SQL" --> db
    sessionRepo -- "SQL" --> db
    deviceRepo  -- "SQL" --> db
    tokenRepo   -- "SQL" --> db

    classDef component fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external  fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef database  fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
```

**Чтение:**

- Входящий gRPC проходит цепочку interceptors (`trace → logging → error mapping → recovery`), затем попадает в один из двух транспортов.
- `AuthTransport` обслуживает `Login / Logout / RefreshToken`, `ManagementTransport` — `CreateUser`.
- Многошаговые записи (Login, RefreshToken, CreateUser) проходят через `Tx Manager.BeginFunc(ctx, fn)` — единая транзакционная граница.
- Репозитории не логируют и сами не открывают транзакции: они получают `pgx.Tx` от `Tx Manager`-а и выполняют SQL.

## Уровень 3 — Components: Account Service `[скелет]`

Структура `account-service` — целевой облик. Код-скелет уже отражает эту топологию (`internal/account-service/internal/...`).

```mermaid
flowchart TB
    gateway[/"Gateway Service<br/><span style='font-size:11px'>gRPC reader<br/>get/list/balance/status</span>"/]:::external
    tx[/"Transaction Service<br/><span style='font-size:11px'>gRPC writer<br/>Debit · Credit</span>"/]:::skeletonExt

    subgraph accSvc["Account Service · :8083 [скелет]"]
        direction TB

        interceptors["gRPC Interceptors<br/><span style='font-size:11px'>chain: trace → logging → error mapping → recovery</span>"]:::component

        subgraph transport_layer["Transport (gRPC)"]
            direction LR
            accountTransport["AccountTransport<br/><span style='font-size:11px'>CreateAccount · GetAccount<br/>GetUserAccounts · GetBalance<br/>UpdateStatus · Debit · Credit</span>"]:::component
        end

        accountService["AccountService<br/><span style='font-size:11px'>Бизнес-логика:<br/>создание счёта · смена статуса<br/>списания/зачисления</span>"]:::component

        subgraph crosscut["Cross-cutting"]
            txManager["Tx Manager<br/><span style='font-size:11px'>pgx · BeginFunc(ctx, fn)</span>"]:::component
        end

        accountRepo["Account Repository<br/><span style='font-size:11px'>pgx · CRUD счетов<br/>атомарный UPDATE balance</span>"]:::component
    end

    db[("Account DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>accounts</span>")]:::database

    gateway -- "gRPC" --> interceptors
    tx      -- "gRPC<br/>Debit/Credit" --> interceptors

    interceptors      --> accountTransport
    accountTransport  -- "domain types" --> accountService

    accountService -- "BeginFunc" --> txManager
    txManager      -- "pgx.Tx" --> accountRepo
    accountRepo    -- "SQL" --> db

    classDef component   fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external    fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeletonExt fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
    classDef database    fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
```

**Чтение:**

- У сервиса **два класса клиентов**: gateway (read-only выборки + создание/смена статуса) и transaction-service (внутренние списания/зачисления).
- Атомарное изменение баланса всегда идёт через `Tx Manager.BeginFunc` — даже для одношагового `UPDATE`, чтобы сохранить транзакционные инварианты при будущих расширениях (несколько проводок в одном вызове).
- Один репозиторий `Account Repository` — модель счёта плоская: `accounts(id, user_id, balance, currency, status, ...)`.

## Уровень 3 — Components: Transaction Service `[скелет]`

`transaction-service` оркестрирует переводы и пополнения: пишет в свою БД, дёргает `account-service` по gRPC, публикует событие в Kafka. Код-скелет в `internal/transaction-service/internal/...`.

```mermaid
flowchart TB
    gateway[/"Gateway Service<br/><span style='font-size:11px'>gRPC client</span>"/]:::external
    account[/"Account Service<br/><span style='font-size:11px'>gRPC :8083<br/>Debit · Credit</span>"/]:::skeletonExt
    kafka{{"Kafka<br/><span style='font-size:11px'>topic: transactions.completed</span>"}}:::infra

    subgraph txSvc["Transaction Service · :8084 [скелет]"]
        direction TB

        interceptors["gRPC Interceptors<br/><span style='font-size:11px'>chain: trace → logging → error mapping → recovery</span>"]:::component

        subgraph transport_layer["Transport (gRPC)"]
            direction LR
            transferT["Transfer"]:::component
            replenishT["Replenish"]:::component
            getT["GetTransaction"]:::component
            historyT["GetHistory"]:::component
        end

        transactionService["TransactionService<br/><span style='font-size:11px'>Оркестрация:<br/>idempotency · BEGIN → Debit → Credit → COMMIT<br/>→ Kafka produce</span>"]:::component

        subgraph clients["Internal Clients"]
            accountClient["Account gRPC Client<br/><span style='font-size:11px'>с trace metadata</span>"]:::component
        end

        subgraph crosscut["Cross-cutting"]
            direction LR
            txManager["Tx Manager<br/><span style='font-size:11px'>pgx · BeginFunc(ctx, fn)</span>"]:::component
            producer["Kafka Producer<br/><span style='font-size:11px'>events после COMMIT</span>"]:::component
        end

        transactionRepo["Transaction Repository<br/><span style='font-size:11px'>pgx · CRUD транзакций<br/>idempotency_key uniq</span>"]:::component
    end

    db[("Transaction DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>transactions</span>")]:::database

    gateway -- "gRPC" --> interceptors

    interceptors --> transferT
    interceptors --> replenishT
    interceptors --> getT
    interceptors --> historyT

    transferT  -- "domain types" --> transactionService
    replenishT -- "domain types" --> transactionService
    getT       -- "domain types" --> transactionService
    historyT   -- "domain types" --> transactionService

    transactionService -- "BeginFunc"  --> txManager
    transactionService -- "Debit/Credit" --> accountClient
    transactionService -. "Produce<br/>after COMMIT" .-> producer

    accountClient -- "gRPC + metadata" --> account
    producer      -. "publish" .-> kafka

    txManager       -- "pgx.Tx" --> transactionRepo
    transactionRepo -- "SQL"    --> db

    classDef component   fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external    fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeletonExt fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
    classDef database    fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
    classDef infra       fill:#F3E5F5,stroke:#6A1B9A,stroke-width:2px,color:#4A148C
```

**Чтение:**

- Точка идемпотентности — `idempotency_key` в `transactions`: повторный `Transfer/Replenish` с тем же ключом возвращает существующую запись, а не дублирует движения.
- Транзакционная граница (`BeginFunc`) охватывает: `INSERT transaction → Debit → Credit → UPDATE status='completed'`. Kafka produce **выполняется после COMMIT** — это «outbox-light» без отдельной таблицы (приемлемо, пока консьюмер идемпотентен).
- `Account gRPC Client` живёт внутри transaction-service — это единственный кросс-сервисный синхронный вызов на уровне доменной логики платформы.

- **Consumer ветка** — единственный источник записей в `ledger_entries`. Это сохраняет журнал как «append-only audit log» и исключает гонки с REST-записью.
- Обе ветки используют один и тот же `LedgerService` и `Entry Repository`, поэтому не нужно вводить отдельный application-сервис.
- `commit offset` Kafka происходит **после** успешного `COMMIT` БД — это даёт at-least-once и в сочетании с идемпотентностью consumer'а (по `transaction_id`) обеспечивает корректность журнала.
