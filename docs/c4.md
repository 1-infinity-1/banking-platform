# C4 Диаграммы

C4 Model — иерархия из четырёх уровней абстракции: Context → Container → Component → Code. Диаграммы ниже охватывают первые три уровня.

---

## Уровень 1 — System Context

Кто взаимодействует с системой и какую роль она выполняет в более широкой среде.

```mermaid
flowchart LR
    customer(["👤 Клиент<br/><span style='font-size:11px'>Вход, переводы, выписки</span>"]):::person
    admin(["👤 Администратор<br/><span style='font-size:11px'>Управление пользователями</span>"]):::person

    banking["🏦 <b>Banking Platform</b><br/><span style='font-size:11px'>Микросервисная банковская платформа на Go:<br/>аутентификация · счета · переводы · бухгалтерский журнал</span>"]:::system

    customer -- "REST/HTTPS<br/>JSON" --> banking
    admin   -- "REST/HTTPS<br/>JSON" --> banking

    classDef person   fill:#E3F2FD,stroke:#1565C0,stroke-width:2px,color:#0D47A1
    classDef system   fill:#FFF3E0,stroke:#E65100,stroke-width:2px,color:#BF360C
```

**Чтение:** клиент и администратор — внешние пользователи; они обращаются к платформе только через REST API (gateway-service на `:8081`). Внутреннее устройство платформы раскрывается на следующем уровне.

---

## Уровень 2 — Containers

Какие исполняемые единицы входят в систему, как они общаются и где хранят данные.

```mermaid
flowchart TB
    customer(["👤 Клиент"]):::person
    admin(["👤 Администратор"]):::person

    subgraph banking["🏦 Banking Platform"]
        direction TB

        subgraph edge["Edge"]
            gateway["<b>Gateway Service</b><br/><span style='font-size:11px'>Go · ogen/OpenAPI<br/>REST API · :8081</span>"]:::container
        end

        subgraph auth_sub["Auth"]
            auth["<b>Auth Service</b><br/><span style='font-size:11px'>Go · gRPC · :8082<br/>JWT · сессии · роли</span>"]:::container
            authDb[("Auth DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>users · sessions · devices<br/>refresh_tokens · roles · permissions</span>")]:::database
        end

        subgraph account_sub["Account [скелет]"]
            account["<b>Account Service</b><br/><span style='font-size:11px'>Go · gRPC · :8083<br/>Счета и балансы</span>"]:::skeleton
            accountDb[("Account DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>accounts</span>")]:::database
        end

        subgraph tx_sub["Transaction [скелет]"]
            transaction["<b>Transaction Service</b><br/><span style='font-size:11px'>Go · gRPC + Kafka · :8084<br/>Переводы и пополнения</span>"]:::skeleton
            txDb[("Transaction DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>transactions</span>")]:::database
        end

        subgraph ledger_sub["Ledger"]
            ledgerGrpc["<b>Ledger Service (gRPC)</b><br/><span style='font-size:11px'>Go · gRPC · :8085<br/>Выписки по счетам</span>"]:::container
            ledgerConsumer["<b>Ledger Consumer</b><br/><span style='font-size:11px'>Go · Kafka consumer<br/>Создаёт проводки</span>"]:::container
            ledgerDb[("Ledger DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>ledger_entries</span>")]:::database
        end

        subgraph infra["Infrastructure"]
            kafka{{"<b>Kafka</b><br/><span style='font-size:11px'>Confluent 7.8 (KRaft)<br/>topic: transactions.completed</span>"}}:::infra
        end
    end

    %% External REST
    customer -- "REST :8081" --> gateway
    admin    -- "REST :8081" --> gateway

    %% Internal gRPC (sync)
    gateway -- "gRPC :8082" --> auth
    gateway -- "gRPC :8083" --> account
    gateway -- "gRPC :8084" --> transaction
    gateway -- "gRPC :8085" --> ledgerGrpc
    transaction -- "gRPC :8083" --> account

    %% Async via Kafka
    transaction -. "produce<br/>transactions.completed" .-> kafka
    kafka -. "consume<br/>transactions.completed" .-> ledgerConsumer

    %% Storage (SQL via pgx/v5)
    auth           -- "pgx/v5" --> authDb
    account        -- "pgx/v5" --> accountDb
    transaction    -- "pgx/v5" --> txDb
    ledgerGrpc     -- "pgx/v5" --> ledgerDb
    ledgerConsumer -- "pgx/v5" --> ledgerDb

    classDef person    fill:#E3F2FD,stroke:#1565C0,stroke-width:2px,color:#0D47A1
    classDef container fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeleton  fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
    classDef database  fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
    classDef infra     fill:#F3E5F5,stroke:#6A1B9A,stroke-width:2px,color:#4A148C
```

**Чтение:**

- **Edge** — единственная точка входа (gateway). Все REST-запросы идут только сюда.
- **Auth / Account / Transaction / Ledger** — четыре доменных сервиса, каждый со своей БД (database-per-service).
- **`[скелет]`** на блоках означает, что код есть как структура (`internal/<service>/`), но бизнес-логика ещё не реализована.
- **Сплошные стрелки** — синхронный вызов (REST/gRPC). **Пунктирные** — асинхронные сообщения через Kafka.
- **gateway → transaction → account** — единственный кросс-доменный gRPC-вызов на уровне платформы (нужен для атомарного списания/зачисления в переводах).

## Уровень 3 — Components: Gateway Service

Из каких компонентов состоит `gateway-service` — самый сложный сервис с fan-out к остальным.

```mermaid
flowchart TB
    client(["👤 Клиент"]):::person

    subgraph gw["Gateway Service · :8081"]
        direction TB

        subgraph mw["HTTP Middleware (ingress chain)"]
            direction LR
            traceMW["Trace MW<br/><span style='font-size:11px'>x-trace-id · x-request-id<br/>→ context.Context</span>"]:::component
            logMW["Logging MW<br/><span style='font-size:11px'>method · path · status · latency</span>"]:::component
        end

        handlers["HTTP Handlers (ogen)<br/><span style='font-size:11px'>Сгенерированы из openapi.yaml<br/>валидация → domain types</span>"]:::component
        services["Services<br/><span style='font-size:11px'>Бизнес-оркестрация:<br/>выбор gRPC-клиента · маппинг ответов</span>"]:::component
        errHandler["Error Handler<br/><span style='font-size:11px'>GatewayHandler.NewError<br/>domain error → HTTP status</span>"]:::component

        subgraph clients["gRPC Clients (с trace metadata)"]
            direction LR
            authClient["Auth Client"]:::component
            accountClient["Account Client"]:::component
            txClient["Transaction Client"]:::component
            ledgerClient["Ledger Client"]:::component
        end
    end

    auth[/"Auth Service<br/>gRPC :8082"/]:::external
    account[/"Account Service<br/>gRPC :8083 [скелет]"/]:::skeletonExt
    tx[/"Transaction Service<br/>gRPC :8084 [скелет]"/]:::skeletonExt
    ledger[/"Ledger Service<br/>gRPC :8085"/]:::external

    client     -- "HTTP/JSON" --> traceMW
    traceMW    --> logMW
    logMW      --> handlers
    handlers   -- "domain types" --> services
    handlers   -. "on error" .-> errHandler

    services --> authClient
    services --> accountClient
    services --> txClient
    services --> ledgerClient

    authClient    -- "gRPC + metadata" --> auth
    accountClient -- "gRPC + metadata" --> account
    txClient      -- "gRPC + metadata" --> tx
    ledgerClient  -- "gRPC + metadata" --> ledger

    classDef person      fill:#E3F2FD,stroke:#1565C0,stroke-width:2px,color:#0D47A1
    classDef component   fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external    fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef skeletonExt fill:#F5F5F5,stroke:#757575,stroke-width:1.5px,stroke-dasharray:5 3,color:#424242
```

**Чтение:**

- HTTP-запрос идёт через цепочку middleware `Trace → Logging → Handlers (ogen)`.
- Handlers вызывают внутренние Services, которые оркестрируют выбор нужного gRPC-клиента.
- Все 4 gRPC-клиента (`authClient`, `accountClient`, `txClient`, `ledgerClient`) инжектируют `x-trace-id` / `x-request-id` в metadata — это даёт сквозную трассировку.
- Ошибки уходят через `Error Handler` (пунктирная стрелка), который маппит domain-ошибки в HTTP статусы.

## Уровень 3 — Components: Auth Service

Внутренняя структура `auth-service` — наиболее полностью реализованного сервиса.

```mermaid
flowchart TB
    gateway[/"Gateway Service<br/><span style='font-size:11px'>gRPC client</span>"/]:::external

    subgraph authSvc["Auth Service · :8082"]
        direction TB

        interceptors["gRPC Interceptors<br/><span style='font-size:11px'>chain: trace → logging → error mapping → recovery</span>"]:::component

        subgraph transport_layer["Transport (gRPC)"]
            direction LR
            authTransport["AuthTransport<br/><span style='font-size:11px'>Login · Logout · RefreshToken</span>"]:::component
            mgmtTransport["ManagementTransport<br/><span style='font-size:11px'>CreateUser</span>"]:::component
        end

        subgraph service_layer["Services (business logic)"]
            direction LR
            authService["AuthService<br/><span style='font-size:11px'>bcrypt · session lifecycle<br/>token generation</span>"]:::component
            mgmtService["ManagementService<br/><span style='font-size:11px'>создание пользователей<br/>назначение ролей</span>"]:::component
        end

        subgraph crosscut["Cross-cutting"]
            direction LR
            tokenMgr["JWT Manager<br/><span style='font-size:11px'>golang-jwt<br/>access/refresh</span>"]:::component
            txManager["Tx Manager<br/><span style='font-size:11px'>pgx<br/>BeginFunc(ctx, fn)</span>"]:::component
        end

        subgraph repos["Repositories (pgx)"]
            direction LR
            userRepo["User Repo"]:::component
            sessionRepo["Session Repo"]:::component
            deviceRepo["Device Repo"]:::component
            tokenRepo["RefreshToken Repo"]:::component
        end
    end

    db[("Auth DB<br/><span style='font-size:11px'>PostgreSQL 17<br/>users · sessions · devices<br/>refresh_tokens · roles · permissions</span>")]:::database

    gateway       -- "gRPC" --> interceptors
    interceptors  --> authTransport
    interceptors  --> mgmtTransport
    authTransport -- "domain types" --> authService
    mgmtTransport -- "domain types" --> mgmtService

    authService   -- "generate/validate" --> tokenMgr
    authService   -- "BeginFunc" --> txManager
    mgmtService   -- "BeginFunc" --> txManager

    txManager -- "pgx.Tx" --> userRepo
    txManager -- "pgx.Tx" --> sessionRepo
    txManager -- "pgx.Tx" --> deviceRepo
    txManager -- "pgx.Tx" --> tokenRepo

    userRepo    -- "SQL" --> db
    sessionRepo -- "SQL" --> db
    deviceRepo  -- "SQL" --> db
    tokenRepo   -- "SQL" --> db

    classDef component fill:#E1F5FE,stroke:#0277BD,stroke-width:1.5px,color:#01579B
    classDef external  fill:#E8F5E9,stroke:#2E7D32,stroke-width:2px,color:#1B5E20
    classDef database  fill:#FFF8E1,stroke:#F57F17,stroke-width:2px,color:#E65100
```

**Чтение:**

- Входящий gRPC проходит цепочку interceptors (`trace → logging → error mapping → recovery`), затем попадает в один из двух транспортов.
- `AuthTransport` обслуживает `Login / Logout / RefreshToken`, `ManagementTransport` — `CreateUser`.
- Многошаговые записи (Login, RefreshToken, CreateUser) проходят через `Tx Manager.BeginFunc(ctx, fn)` — единая транзакционная граница.
- Репозитории не логируют и сами не открывают транзакции: они получают `pgx.Tx` от `Tx Manager`-а и выполняют SQL.
